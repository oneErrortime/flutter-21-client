A complete Android voice call application built on WebRTC with DTLS-SRTP end-to-end encryption. The signaling server never touches your audio — all media flows peer-to-peer, fully encrypted.

What is VoiceCall?

VoiceCall is a Flutter Android app paired with a Node.js signaling server (and an alternative Rust server) that enables direct peer-to-peer voice communication. The server only brokers the connection setup — once established, audio travels directly between devices, encrypted with DTLS-SRTP.

🔒
End-to-End Encrypted
DTLS-SRTP (RFC 5764) — the server cannot decrypt your calls
⚡
P2P Direct
ICE/STUN traversal — audio bypasses the server entirely
🔄
TURN Fallback
Relay mode for corporate NAT / symmetric firewalls
🔑
JWT Auth
Access tokens + rotating refresh tokens with reuse detection
🔗
Deep Links
Share room links — voicecall://join/ROOM
🦀
Rust Server
Alternative high-performance Rust signaling backend

Architecture

VoiceCall uses a classic WebRTC architecture: a lightweight signaling server handles connection negotiation, while all audio flows directly peer-to-peer.

System Diagram
┌──────────────┐         WSS/HTTPS          ┌─────────────────────┐
│ Android App  │ ◄──────────────────────── ► │  Signaling Server   │
│  (Flutter)   │     JWT-authenticated       │  (Node.js + WS)     │
└──────┬───────┘         WebSocket          └────────┬────────────┘
       │                                              │ MongoDB
       │         Direct P2P (ICE/STUN/TURN)           │
       └──────────────────────────────────────────────┘
                  DTLS-SRTP encrypted audio

Two calling modes

ModeHow it worksWhen to use
P2P (STUN) Direct connection via STUN NAT traversal Same network / simple NAT
P2P (TURN relay) Audio relayed via TURN server Symmetric NAT / corporate firewalls
ℹ️
The server never handles audio. It only brokers the WebRTC handshake (SDP offer/answer and ICE candidates). Once the P2P connection is established, the signaling server is out of the loop entirely.

RFCs & Standards

VoiceCall implements and builds on the following open standards:

RFC 8825 — WebRTC RFC 8829 — JSEP RFC 8445 — ICE RFC 5389 — STUN RFC 8656 — TURN RFC 5764 — DTLS-SRTP RFC 4566 — SDP RFC 7519 — JWT RFC 6749 — OAuth2

Quick Start — Server

Prerequisites

  • Node.js 18+
  • MongoDB (local or Atlas free tier)
  • Docker + Docker Compose (optional)

Option A — Local development

bash 
cd server
cp .env.example .env
# Edit .env — set MONGODB_URI and JWT secrets

npm install
npm run dev

Server starts at http://localhost:3000. WebSocket at ws://localhost:3000/ws.

Option B — Docker (recommended for production)

bash 
cd server
cp .env.example .env
# Edit .env — strong JWT secrets, MongoDB URI, TURN credentials

docker-compose up -d

Generate JWT secrets

bash 
# Generate two separate secrets
openssl rand -hex 64  # JWT_ACCESS_SECRET
openssl rand -hex 64  # JWT_REFRESH_SECRET

WebRTC & P2P

Under the hood VoiceCall uses flutter_webrtc to manage RTCPeerConnection, ICE candidate gathering, and DTLS handshake. The signaling server acts as a rendezvous point, relaying SDP offers/answers and ICE candidates.

Call flow

Caller                    Signaling Server               Callee
  │                             │                          │
  │── auth (JWT) ──────────────►│◄──────────── auth (JWT) ─│
  │                             │                          │
  │── call { offer: SDP } ─────►│── call-incoming ────────►│
  │                             │                          │
  │◄──────────────── call-answered { answer: SDP } ────────│
  │                             │                          │
  │── ice { candidate } ───────►│── ice ──────────────────►│
  │◄─────────────────────────── ice { candidate } ─────────│
  │                             │                          │
  │◄══════════ DTLS-SRTP encrypted P2P audio ══════════════►│

Security

Authentication

FeatureDetails
JWT access tokens15-minute expiry
Refresh tokens30-day expiry, single-use rotation
Reuse detectionAll sessions invalidated on reused refresh token
BcryptPassword hashing with cost factor 12
Rate limiting10 auth attempts / 15 min / IP

Call encryption

🔒
DTLS-SRTP (RFC 5764) — All audio is end-to-end encrypted. WebRTC enforces DTLS by default. The signaling server cannot decrypt calls. DTLS fingerprints are displayed in-app for manual verification (TOFU — Trust On First Use).

Android security

  • Encrypted SharedPreferences via flutter_secure_storage (Android Keystore)
  • Certificate pinning template in network_security_config.xml
  • No cleartext traffic to production domains
  • allowBackup="false" — prevents credential extraction via ADB backup

Server security

  • Helmet.js security headers + CORS whitelist
  • Rate limiting per endpoint
  • MongoDB injection prevention (Mongoose)
  • Payload size limits (10 KB)
  • Non-root Docker container
  • WebSocket authentication before any operations

fingerprint
lt-cred-mech
user=turnuser:strongpassword
cert=/path/to/cert.pem
pkey=/path/to/key.pem
log-file=/var/log/coturn/turn.log
💡
Free alternative: Metered.ca has a free TURN tier. Update lib/core/constants.dart with your TURN credentials.

Deep Links & Call by Link

Users can create shareable room links that expire after 24 hours:

uri 
# Native deep link
voicecall://join/ROOM-UUID

# HTTPS universal link
https://yourapp.com/join/ROOM-UUID

The room creator waits as host. The joiner sends the WebRTC offer when they arrive. Links auto-expire after 24 hours.

Call by User ID

Each user has a unique UUID shown on the home screen. Copy & paste to call directly — no link sharing needed.

Deployment Checklist

  • ☐ Generate strong JWT secrets (openssl rand -hex 64)
  • ☐ Set up MongoDB Atlas or self-hosted MongoDB with auth
  • ☐ Deploy Coturn or configure Metered.ca TURN
  • ☐ Get SSL certificate (Let's Encrypt via certbot)
  • ☐ Update nginx.conf with your domain
  • ☐ Update constants.dart with production URLs
  • ☐ Update AndroidManifest.xml deep link domain
  • ☐ Update network_security_config.xml domain
  • ☐ Set ALLOWED_ORIGINS to your app's domain
  • ☐ Enable android:usesCleartextTraffic="false" for release
  • ☐ Consider certificate pinning in network_security_config.xml
  • ☐ Test calls on real devices across different networks
